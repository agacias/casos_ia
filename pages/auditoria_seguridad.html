Auditoría de Seguridad

Meta-prompt

dame un prompt para hacer un analisis de seguridad

Prompt

"Actúa como un Senior Security Engineer (Pentester) especializado en auditoría de código estático (SAST). Analiza este proyecto en busca de vulnerabilidades críticas basándote en el OWASP Top 10 y las mejores prácticas de seguridad de 2026.

Por favor, enfócate en:

  • Inyecciones: Busca puntos de entrada propensos a SQL Injection, XSS, Command Injection o NoSQL Injection.
  • Gestión de Secretos: Identifica claves de API, tokens o contraseñas 'hardcoded' o expuestas en archivos de configuración.
  • Autenticación y Autorización: Revisa fallos en el control de acceso, sesiones inseguras o lógica de permisos débil.
  • Fugas de Datos: Detecta si se está registrando información sensible (PII) en los logs o si se envía información innecesaria en las respuestas de la API.
  • Configuraciones Inseguras: Revisa cabeceras HTTP, políticas de CORS y configuraciones de entorno.

Formato de salida: Presenta los hallazgos en una tabla con: Severidad (Crítica, Alta, Media, Baja), Archivo:Línea, Descripción del riesgo y Código sugerido para mitigarlo."

2. Prompts Específicos (Seguridad por Capas)

Si quieres profundizar en áreas concretas, usa estas variaciones:

A. Para "Limpieza" de Secretos y Variables

"Escanea el código y los archivos de configuración en busca de cualquier cadena de texto que parezca un secreto, clave privada, token JWT o credencial. Verifica también si el archivo .gitignore está configurado correctamente para excluir entornos sensibles (.env, .pem, .json de credenciales)."

B. Para Seguridad en el Flujo de Datos (Backend)

"Analiza cómo se procesan las entradas del usuario desde los controladores hasta la base de datos. Identifica si existe sanitización adecuada y si se están utilizando consultas parametrizadas o ORMs de forma segura para evitar ataques de inyección."

C. Para Auditoría de Dependencias (Supply Chain)

"Revisa el archivo de gestión de dependencias (ej. package.json, requirements.txt, go.mod). Identifica librerías con vulnerabilidades conocidas o versiones obsoletas. Sugiere actualizaciones seguras y evalúa el riesgo de ataques de confusión de dependencia."

3. Visualización del Proceso de Seguridad

Un análisis de seguridad robusto con IA no solo encuentra errores, sino que ayuda a implementar un ciclo de vida de desarrollo seguro (SDLC).

4. Tabla de Clasificación de Riesgos (Output sugerido)

Puedes pedirle a la IA que use este formato para que los resultados sean procesables por tu equipo:

Vulnerabilidad Ubicación Impacto Recomendación
SQL Injection db/users.js:L45 Crítico: Acceso total a la DB Usar Prepared Statements.
Sensitive Data Leak utils/logger.js:L12 Medio: Exposición de PII Enmascarar el email en logs.
Missing CSRF routes/auth.js Alto: Secuestro de sesión Implementar tokens CSRF.

Consejos para un mejor resultado:

  • Contexto del Framework: Añade al inicio del prompt: "Este es un proyecto de [Node.js/Django/Spring Boot]". La IA buscará patrones de ataque específicos de esa tecnología.
  • Pide el "Exploit" (Opcional): Si no entiendes el riesgo, dile: "Explícame cómo un atacante podría explotar esta vulnerabilidad en este código específico". Esto es excelente para el onboarding de desarrolladores junior.
  • Modo Paranoico: Puedes añadir: "Sé extremadamente riguroso, incluso con las vulnerabilidades teóricas o de baja probabilidad".
